## Issues Found and Fixed

### 🔴 Issue 1 — `safePath()` prefix-match path traversal (SECURITY)
**File:** `routes/fileRoutes.js`  
**Severity:** High  

**Problem:**  
The original check was:
```js
if (!resolved.startsWith(path.resolve(root))) {
    return null;
}
```
This fails when a project name is a prefix of another project name. Example:
- Project `Foo` → root = `/storage/Foo`
- Sibling project `FooBar` → root = `/storage/FooBar`
- Traversal: `source = "../FooBar/evil"` → resolves to `/storage/FooBar/evil`
- Check: `"/storage/FooBar/evil".startsWith("/storage/Foo")` → **`true`** (passes!)

An authenticated user with access to project `Foo` could read/write files from any project whose name starts with `Foo`.

**Fix applied:**
```js
const resolvedRoot = path.resolve(root);
if (resolved !== resolvedRoot && !resolved.startsWith(resolvedRoot + path.sep)) {
    return null;
}
```
Adding `+ path.sep` ensures `/storage/Foo` only matches paths that are literally inside `/storage/Foo/`, not siblings sharing its prefix.

**Verified:** Unit-tested before/after — sibling traversal now blocked, all legitimate paths still pass.

### 🟡 Issue 2 — Dead 410 code path in `serveShare` (BUG)
**File:** `routes/fileRoutes.js`  
**Severity:** Medium (UX — expired shares always show "not found" instead of "expired")

**Problem:**  
`serveShare` opened with:
```js
const data = await cleanExpiredShares();  // ← removes expired entries from data
const share = data.shares[shareId];

if (!share) {
    return res.status(404).send(...);
}
if (new Date(share.expiresAt) <= new Date()) {  // ← dead: share already deleted above
    return res.status(410).send(...);
}
```
`cleanExpiredShares()` deletes expired entries from the returned object (and writes to disk). So any expired share is already gone when we check `data.shares[shareId]` → falls through to 404. The 410 branch could never be reached.

Users visiting an expired link would see "This link doesn't exist or has been removed" instead of the helpful "This link has expired" message.

**Fix applied:**  
Changed to `readShares()` (no cleanup) first, check expiry inline, then fire-and-forget cleanup:
```js
const data = await readShares();           // raw read, no cleanup
const share = data.shares[shareId];

if (!share) {
    return res.status(404).send(...);
}
if (new Date(share.expiresAt) <= new Date()) {
    cleanExpiredShares().catch(() => {});  // fire-and-forget
    return res.status(410).send(...);
}
```

## Overall Assessment

**The cloud features are well-implemented and production-ready after the two fixes.**

The copy/move/share architecture is solid: paths are validated, operations are atomic where possible, metadata is tracked, and the public share endpoint is cleanly isolated from the authenticated API. The frontend folder picker is fully functional with proper disabled states for invalid destinations (move-into-self, move-to-same-location).

Two bugs were patched:
1. A **security vulnerability** in `safePath()` (prefix-match traversal) that could allow cross-project file access between projects sharing a name prefix.
2. A **UX bug** in `serveShare` where expired links always showed "not found" instead of the intended "link expired" page.

No other issues found.
